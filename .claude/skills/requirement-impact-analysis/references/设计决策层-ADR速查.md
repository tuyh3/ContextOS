# 设计决策层速查:Evidence-grounded ADR-lite(子项目 ②)

> Phase 3.5 设计综合的机器。SKILL.md 指向本文件;输出形态(报告里 §3 怎么排 + worked example)看 `输出模板.md`。
> 一句话:看到主数据归并气味 -> 开 ADR -> 推荐一个架构方向([设计]),但 grounded 在现状([事实])+ 标清推断([推断])+ 给兼容边界 + 给翻案条件,绝不借现状段可信度。

## 1. 何时开 ADR(D5 触发:目录优先 + fork 自检 + 影响阈值)

不是每个需求都开 ADR(守"别教条化")。按顺序判:

### 1.1 design-smell 目录(命中即候选开 ADR)

- **主数据归并**(已被真盲测验证):静态字典表 + CODE_TYPE 类字段 + 多消费方 + 前端前缀/分类逻辑散落。-> 推"统一主数据实体 + 迁移 + 读点替换 + 兼容边界",而非散落补丁表。
- **生命周期**(起始 hypothesis):需求引入 启用 / 停用 / 下架 / 审批 / 审计 / 状态流转。
- **跨需求共享实体**(hypothesis):多需求点操作同一概念实体 / 同一规则集。
- **分类机制替换**(hypothesis):从字符串 / 前缀 / 枚举散落逻辑 升级为显式类型字段。
- **持久化归属**(hypothesis):需新增表 / 扩旧表 / 迁旧数据 / 定 owner schema。

注:目录可调,只有"主数据归并"被真盲测坐实,其余是起始 hypothesis,随真跑去重 / 补。
注:**`兼容边界` 不是触发器** —— 它是每个 ADR 的必写段(见 §4),几乎所有老系统需求都有,单独作触发会过度开 ADR。

### 1.2 目录没中 -> 架构 fork 自检(任一为是即候选)

- 存在 2+ 合理落点(放哪都说得通)?
- 改变长期数据模型?
- 影响多 caller / 多页 / 多批 / 多接口?
- 涉历史数据迁移 / 兼容格式?
- 选错会大面积返工?

### 1.3 影响阈值(防噪音,读 ① 的真 blast-radius,非空想)

读 ① 给的真计数(`lookup_calls` callers / `search_source` consumers / 受影响 tables)。返工面**局限单方法 / 单页 / 单 SQL** -> **不开 ADR**(保持纯定位器 X.1-X.4)。

### 1.4 都不中 = 纯定位器

该需求保持现有 X.1-X.4 形态,不强加设计决策。

## 2. 三标签(D4:ADR 段内严格三分)

ADR 段内只用这三个,修"把推断/设计伪装成事实":

- `[事实]` = ① 定位的源码 / DB / SQL / 调用链证据(可点开核,带 MCP 出处)。
- `[推断]` = 从事实推出的**系统含义**(defeasible,可被新证据推翻;如"该静态字典表正在承担业务主数据角色")。**这是 ② 新增的关键标签** —— 旧系统无处安放它,就被塞进了 `[事实]`。
- `[设计]` = 真正的**架构选择**(如"新增统一主数据表")。
- **`[推断]` vs `[设计]` 判别**:对**现状**的解读("系统现在其实在做 X")标 `[推断]`;提出**现状没有的**新结构 / 新动作("应新增 / 改成 Y")标 `[设计]`。两者都**不可点开核**,故都绝不是 `[事实]`。

铁律:设计器**可以推荐**(`[设计]`),**不许伪装成证据**(`[事实]`),中间的解释明确标 `[推断]`、可逐环挑战。段外(逐需求块 X.1-X.4)保持现有 `[事实]/[建议]/gap/[背景]/[背景-gap]` 不变。

## 3. ADR-lite 空模板(D3:逐段填,缺必写段 = 不算 ADR)

每个决策 = 一个 ADR-lite block,固定结构:

```
### 决策 N:<标题>
[设计] <一句话推荐方向>

**现状证据**(引①,带 MCP 出处)
- <[事实] ...> 出处:<工具 + 参数 + 命中摘要>

**推断链**(从事实推出的系统含义,defeasible)
- [推断] <如"该静态字典表正在承担业务主数据角色">(依据上列事实;可被新证据推翻)

**候选方案**(>= 2)
1. <方案 A>  2. <方案 B>  3. <方案 C>

**推荐 + 理由**
[设计] 选 <方案 X>,因 <理由,grounded 在现状/推断>。

**被否方案 + 为什么否**
- 否 <方案 Y>:因 <理由>。

**概念模型与范围意图**(D1 概念级;禁可实施级)
- 概念实体:<需要什么实体>;关键语义字段:<类型/状态/生命周期/审计 的含义,非 DDL>。
- 接口/页面/权限范围意图:<需要新 X 接口、新 Y Tab —— 范围,非签名/布局>。
- 迁移策略意图:<从 Z 迁来,保留 W 兼容 —— 方向,非迁移 SQL>。

**兼容边界**(不变量,必写段)
- <历史表/编码/前缀/接口入参/页面 tab 必须保留的不变量>。

**证据依赖与降级**(强制表,D6)
| Claim | 依赖证据 | 当前状态 | 影响 |
|---|---|---|---|
| <方向 claim> | <lookup_table schema + 滥用证据 + 需求语义> | complete | 可推荐 |
| <方向 claim, 但 lookup_table 未跑> | <lookup_table offline 列空> | schema 未验证 | 不许推荐, 仅列候选 |
| <范围 claim> | <consumer census = search_source/lookup_calls> | truncated | 不得声称范围闭合(不连累方向) |

**翻案条件**
- 若 <什么证据出现> -> 改选 <别的方案>。

**置信度**:ADR 总体 <高/中/低>;关键 claim 级见上表"影响"列。
```

## 4. 四 gate(D8:模板必写段,缺即不算 ADR)

- **候选方案 gate**:凡"新表 vs 扩旧 / 统一模型 vs 散落补丁"架构点,至少列 2 候选 + 1 被否理由(缺 -> 不算 ADR)。
- **兼容边界 gate**:涉历史表 / 编码 / 前缀 / 接口入参 / 页面 tab,必单列不变量(必写段)。**现状定位到的有约束力约定按 D12(§10)升为不变量**。
- **翻案条件 gate**:每个推荐必写"什么证据出现就改选别的方案"。
- **降级 gate**:每个 ADR 必带证据依赖表,**按证据源分别判**(见 §5)。

注:四 gate 的"模板含必写段"由测试结构探针机械守(`test_requirement_skill_neutral`);"段内填得对不对"靠人工 gold 审。机械守"没被抽空"、人工守"填得对"。
注:四 gate 是**下限**,不是必写段全集 —— §3 模板的 **概念模型与范围意图 / 置信度** 同样必填(只是没单设 gate),别因"只 gate 了 4 项"就省略它们。

## 5. 降级(D6:per-claim 粒度,按证据源分)

降级不是整个 ADR 降,是 **claim 粒度 + 按证据源**:

- **方向 claim**(新表 vs 扩旧)依赖 现有表结构(`lookup_table`)+ 需求语义 + 滥用证据。
  - consumer census(`search_source`/`lookup_calls`)截断**不影响方向**(方向不靠"全量读点枚举"),仍**可推荐**。
  - 但 **`lookup_table` 未跑 / offline 列空 / 失败** -> 表级方向(扩旧 vs 新表)schema 证据缺 -> **不许推荐具体方向,只列候选 + 标 `[限制] schema 未验证`**(可保留"需要主数据实体"的 `[推断]`,不下"扩旧/新表"判决)。
- **范围 claim**(替换全部读点 / 完整影响面)只依赖**完整** consumer census -> `search_source` `truncated=true` / `lookup_calls` 未跑 / 关键 census 红灯 -> 标 `[限制] 范围未闭合,③ 阶段补全`,**不降整个 ADR、不连累方向 claim**。

大仓样本实测 `search_source` 常截断(`files_scanned` 触 cap、`truncated=true`),所以范围 claim 这条**真会触发** —— ② 因此"推荐方向 + 诚实标范围未闭合",不哑掉也不硬称完整。

## 6. 概念级 vs 可实施级(D1:② 的天花板,过线即 ③)

- **② 出(概念级)**:概念实体 / 关键语义字段含义(类型 / 状态 / 生命周期 / 审计 的**业务含义与为什么需要**,非 DDL)/ 接口·页面·权限**范围意图** / 迁移**策略意图** / 兼容边界 / 翻案条件。
- **② 禁(可实施级 = ③)**:精确 DDL(列名 / 类型 / 精度 / NOT NULL / 索引)/ 完整 `@Service` inMsg/outMsg 契约 / 页面布局 / 迁移 SQL / DAO 实现。

边界线一句话:
- ②:"需要一个统一主数据实体,含 类型/状态/生命周期/审计 语义,因现状缺这些。"
- ③:"`BASE.<表>(<列> NUMBER(2) NOT NULL, ... 索引 idx_...)`。"

② 给决策 + 概念骨架让人判断对不对;③ 给可建的物。

## 7. X.3 设计就近 + 逃逸口封死(D7)

逐需求块 X.3 是"本需求设计思路",必须让读者在本需求块内直接看懂设计方向;但它**不得独立提出架构落点**。

规则:
- 若本需求命中 §3 ADR:X.3 写本需求可读摘要 + 引用 `§3 决策 N`;**不能只写"遵循 决策 N"**。
- 凡 X.3 `[建议]` 涉及 数据模型 / 迁移 / 接口边界 / 页面边界 / 兼容格式 -> **必须受 §3 对应 ADR 约束**。
- **commit vs 设计空间按证据强弱分(非需求数量)**:证据足(方向有现状/推断硬撑)-> 开 §3 完整 ADR(单/跨需求皆可)+ X.3 committed 摘要;证据薄(关键依赖缺 / census 截断)-> X.3 写"**设计空间 / 待设计决策**"(列候选 + 缺什么证据),不许就地硬下架构判断。

目的:既避免读者在逐需求块和 §3 之间来回跳,又防模型在 X.3 用 `[建议]` 偷做本该进 ADR 的架构决策。

## 8. 落点归属表(D10:把 §3 决策下沉约束逐需求块落点)

S1' 病:②会开 §3 ADR、命中统一主数据方向,但决策不下沉 —— 各需求块 X.3/X.4 仍各自散落建表/config,报告内部分裂。D10 用一张**条件触发**的横切「落点归属表」收口。

### 8.1 触发条件

仅当某个 §3 决策**同时约束 2+ 需求块 或 2+ 落点/范围点**时建表。

- "落点/范围点" = **存储 / 状态 / 配置 / 接口 / 页面** 五类。
- "或"不是"且":既咬跨需求块散裂(同一决策的落点分散在 R1/R3),也咬单需求块内部散裂(一块里 2+ 落点归同一决策)。
- 纯单点设计 / 纯定位器 / 简单需求(单方法单页单 SQL)**不建表**(守 §1.3 影响阈值 + "别教条化")。

### 8.2 表结构(列固定)

| 落点 ID | 落点/范围点(存储/状态/配置/接口/页面) | 出现于需求块 | 归属 ADR | conform? | 说明 |
|---|---|---|---|---|---|

- `落点 ID`:`L-01` / `L-02` ...(给 X.3 摘要 + X.4 落点明细引用 + 探针锚点)。
- `conform?`:**固定三值枚举,不许自由写** —— `conform` / `翻案 -> §3 决策 N` / `gap/待确认`。

### 8.3 铁律:架构级落点不得无背书出现

- **架构级落点/范围点**(新表 / 新 config 载体 / 新状态源 / 新接口边界 / 新页面边界)在 X.3 摘要 和 X.4 落点明细里**一律不得无背书出现**,必须挂到一个 §3 决策。挂法按"是否触发 D10 进表"分流:
  - 该决策**已触发 D10 进表** -> X.3 摘要 + X.4 明细都标**同一个 L-ID**。
  - 该决策**单需求单落点、未触发 D10 表** -> X.3/X.4 引用 `§3 决策 N`(**无 L-ID**,因无表),仍不许内联默认架构落点。
- **X.4 不得新增"既无 L-ID(表存在时)也无 §3 决策 N 背书"的架构级落点/范围点**;非架构级存量改点(改某 FQN 一行 / 加一个 callers)不受此约束。

### 8.4 引用硬格式(固定句式,防散)

- X.3(进表时):`本需求落点受 §3 决策 1 约束,归属见 §4.x 落点归属表 L-02。`
- X.3(未进表单点 ADR):`本需求落点受 §3 决策 N 约束(单点,未入归属表)。`
- X.4 落点明细行:进表标 `(归属 L-02)`,未进表标 `(背书 §3 决策 N)`,后缀挂在该架构级落点行。
- 翻案必须**另开一个 §3 ADR**,表里标 `翻案 -> §3 决策 N`(或暂不能定标 `gap/待确认`),不许 X.3/X.4 就地硬拐。

## 9. 架构 fork 不得 X.3/X.4 内联判决(D11:开不开 ADR 按 D5、进不进表按 D10)

§1 已有 "fork 自检",D11 把它对 X.3/X.4 收口成硬规则,但**三段解耦**,防与 §1.3 降噪、§8.1 触发同时打架:

- X.3/X.4 一旦涉及 数据模型 / 新建表 vs 扩旧 / 迁移 / 接口边界 / 页面边界 的**选择**(2+ 合理落点/范围点),**架构 fork 不得 X.3/X.4 内联判决** —— 不许在 X.3 摘要或 X.4 落点明细里内联敲定一个默认 config/表/接口/页面 溜过。
- **是否开 ADR 按 D5 影响阈值**(§1.3):返工面够大 / 证据够 -> 开 §3 ADR;**返工面局限单方法/单页/单 SQL 或证据薄** -> X.3 只写"**设计空间 / 待决策**"(列候选 + 缺什么证据),既不强开 ADR,也不内联判决。
- **是否进落点归属表按 D10 触发**(§8.1):只有"已开 ADR 且约束 2+ 需求块或 2+ 落点/范围点"才进表(接口/页面 fork 同此口径);单需求单落点的 ADR 开了即可,不强塞横切表(此时 X.3/X.4 引用 `§3 决策 N`、无 L-ID,见 §8.3)。

一句话:**内联禁(无条件)/ 开不开 ADR 看 D5 / 进不进表看 D10**,三件事别混。

## 10. 兼容不变量清单(D12:现状定位到的约定升为不变量,按已开 ADR 作用域)

现状 drill 定位到的编码 / 前缀 / 格式约定,不能只在现状段一提了之,必须**升为不变量**喂进对应 ADR 的「兼容边界」段(§4 已是必写 gate,D12 强制把"定位到的约定"喂进去)。

- **作用域(防全局误伤)**:**按已开 ADR**,非全局。对每个已开启的 §3 ADR,凡**约束该 ADR 方案选择 / 迁移兼容**的现状编码 / 前缀 / 格式约定,必须进该 ADR 的兼容边界;漏一条即 fail。无 ADR 的需求不受此约束(ADR 是 Phase 3.5 才决定开不开,自检落 Phase 3.5 / Phase 4)。
- **限制词(防 ADR 膨胀)**:只升**对方案选择或兼容迁移有约束力**的约定(如 `<前缀>_+id` 类影响主键 / 路由的命名规则、报文格式、菜单 serviceCode 规约);**普通命名风格不升**(变量名 / 类名风格不进不变量)。
