# 输出模板:概要设计初稿(两层:业务正文 + 溯源附录)

报告是业务专家式的概要设计(总-分-总),分**两层单文件**:

- **业务正文**(前):从需求分解读到横切汇总,一气呵成,像专家文档 —— **零任何 bracket 装置**(`出处:` / 逐行 `[事实]/[建议]/[推断]/[设计]/[背景]/[限制]` / `(归属 L-NN)` / `[gap]` / `[冲突]` / `[背景-gap]`)。存量名 / 落点旁挂脚注号 `[^aN]`。开放项与矛盾**用大白话**点名(未定位 / 原文未给 / 人·PD / ③ 四类 + 谁来定;矛盾用"重要:...不一致"),不留尖括号记号 —— 记号给审计者,业务方读不懂(见"四条硬规则·开放项与矛盾")。
- **溯源附录**(后,`==== 溯源附录 ====` 之后):逐条出处(脚注定义)+ 落点归属表 + 证据降级 + MCP 调用清单。给要核验的人;业务方可略。

**铁律:可重放核验性一字不丢,全在附录 —— 这是降级 / 搬家,不是删。** 证据是 skill 的护城河(它不是权威、是 grounded 定位器),但放错层会让业务方读不下去,所以从正文降到审计层。

整份报告写入 **gitignored** 路径(客户 FQN/表名不进 tracked 文件)。本文件末尾的 worked example 是**中性合成**(`com.example.app.*` / `APP_*`),仅作格式示范。

## 业务正文 = 专家骨架 + 两 delta

业务正文直接采用业务专家概要设计骨架,只与专家差两点:

- **delta 1:止于概念级。** 专家骨架里 数据模型设计(DDL 列清单)/ 接口设计(java 签名体)/ 前端设计(页面布局)/ 具体源码文件清单 / 风险评估 / 测试建议 / 实施计划(工期)/ 迁移 SQL,skill **不产**(那是 ③ 详细设计 / 人·PD 的活)。skill 到"该建什么实体 + 含什么语义字段 + 照哪个邻居 + 兼容边界"为止。
- **delta 2:末尾加溯源附录。** 专家不需要(他是权威),skill 需要(可核验定位器)。

专家章节 -> skill 在此产什么(概念级):

- 需求分解(轻列表,几行枚举 N 点)-> 报告头区一个轻列表,不单开重章。
- 需求 N(每需求一章,自包含)-> 含四节:当前系统现状分析 / 新需求内容 / 概要设计(设计思路)/ 影响落点。
- 设计决策(跨需求)-> 设计决策章(决策块;跨需求综合放这,单需求块放不下时也在此)。
- 影响评估(全局)-> 横切汇总(跨需求影响盘点 + 全局 gap)。
- 附录 -> 换成溯源附录(A/B/C/D)。

> 系统现状不单开一章,搬进每个需求章的现状节(每需求章自包含;共享事实可在相关需求章各出现一次,换业务自包含可读,不强求 DRY —— 但同一事实跨段给出不一致值时用"重要:...不一致"提示,见横切 4.1)。

## 四条硬规则

### 现状总表(每需求章现状节开头)

一张三列表,后接痛点散文:

| 现状结论 | 涉及对象 | 对设计的含义 |
|---|---|---|
| <人话现状> | <类/表/配置名> [^aN] | <这条现状逼出什么设计决策> |

- 第三列「对设计的含义」是信任闸桥 —— 把现状直接接到决策,业务方一眼核"你对现状理解对不对"。
- 涉及对象名挂脚注 `[^aN]` 到附录 A;表本身不含内联出处 / 标签。
- 没定位到写"未定位到,需进一步核实"(红旗),绝不臆造;"系统无 X"须经负判自查(根因 E)后方可,否则写"未定位到,非确认无"(大白话,不用记号)。

### 影响落点三桶(每需求章影响落点节固定)

固定三个子标题,替代旧的逐行 `[事实]/[建议]`:

- **改动现有**:现有页面 / 方法 / 表 / 批处理入口 / 校验逻辑。
- **新增**:新增表 / 页面 / 接口 / 日志 / 配置。
- **不改但需兼容**:旧表 / 旧字段 / 旧编码规则 / 旧前缀规则(操作化 D12 兼容不变量 —— 兼容项若没这桶,会被误写成新增或漏掉)。

落点行可挂 `[^aN]` 到附录 A(存量改点引证据;纯新增提案引邻居或无脚注)。架构级落点的"归哪个决策"在附录 B 落点归属表,不在正文内联。

### 决策块限长(每个决策 ≤6 段)

专家式干净块,固定六段、不写成论文(三标签 `[事实]/[推断]/[设计]` / 推断链 / 证据依赖降级表全部不在业务决策块 —— 沉附录;verbose ADR 机器模板见 `references/设计决策层-ADR速查.md`):

- **决策**:一句话。
- **为什么**:2-4 条现状原因。
- **方案比较**:只列主候选(纳候选 + 被否)。
- **采纳**:一句话(+ 为何否其他)。
- **影响**:改哪里、新增哪里。
- **兼容/翻案条件**:简短列(纳兼容边界 + 翻案)。

### 开放项与矛盾用大白话(业务正文零 bracket 装置)

`[gap]`/`[冲突]` 这类尖括号记号是 jargon 装置,业务方读不懂;更深的问题是 `[gap]` 语义已被重载掏空(同时盖 4 种本质不同、接手人/严重度都不同的情况)。所以业务正文**不留任何尖括号记号**,开放项**按 4 类各自用大白话点名 + 点明谁来定**:

- **未定位**(工具查了没找到,可能遗漏)-> "未定位到 X(需进一步核实,详见溯源附录)",**不等于不存在**。
- **原文未给**(需求文档没写)-> "需求未明确 X,需**需求方**补充"。
- **人/PD 专属**(枚举编码/索引/字段长度/验收/风险/工期)-> "X 由**产品/PD**确定",这是**正常分工不是缺陷**。
- **③ 详细设计阶段**(精确列名/状态机/页面布局)-> "X 留**详细设计(③)**定"。

矛盾(原 `[冲突]`)用 plain 但醒目:`重要:X 在两处记录不一致,需以 Y 为准(详见溯源附录 C)`。记号 `[冲突]` 沉附录。

> `[gap]` 仍是 skill **内部 drill 阶段**(Phase 0-3.5 authoring)的反幻觉纪律,只是**最终报告业务正文**渲染成大白话;开放项在横切"待确认 / 开放项"小节(plain)兜底,防被静默丢。

## 报告结构(业务正文)

### 0. 报告头 / 元信息

- 输入来源(eml/docx + 文件名)、分析日期、目标 codebase。
- 工具态:`health_check` 结论 + 哪维降级;`profile_info` 的 `rag_corpora`(D9 业务校准白名单)。
- 怎么读:正文干净无任何尖括号记号,开放项 / 矛盾用大白话(见四条硬规则),核验看末尾溯源附录。
- recall 声明(拆两句):需求分解覆盖原文编号(Phase 1 保证);影响落点是已定位证据子集,不保证完整召回。
- MCP 调用清单见附录 D。

### 1. 需求分解

按原文真实层级拆 N 个编号需求点,各一句话 + 可溯源点号(如 `账务-2.A`);标点间耦合。

### 2. 逐需求章(主体,每需求一章四节)

**当前系统现状分析**:现状总表(三列)+ 痛点散文。全 `[事实]` 级证据,回溯到数据源头;脚注挂附录 A。
**新需求内容**:需求描述 + 业务规则(编号)+ 只复述原文验收(没给就写"需需求方补充");业务语义经 RAG 校准(命中 / 未命中都落附录 A 脚注,不在正文贴 `[背景]`)。
**概要设计(设计思路)**:紧跟需求理解写本需求怎么设计(3-6 条短 bullet:推荐方向 / 为什么[用现状解释] / 数据模型意图[概念语义,不写 DDL] / 接口·页面范围意图[不写签名体/布局] / 迁移兼容边界 / 待定项:大白话点名谁来定)。命中设计决策章 ADR 时写本需求摘要 + "遵循 决策 N",**不能只写"遵循 决策 N"** 让读者跳走;证据薄时写"设计空间"列候选不硬推荐。
**影响落点**:三桶(改动现有 / 新增 / 不改但需兼容)。改存量给落点 + 改法方向;全新增给方向 + 照哪个邻居,**不写 schema/接口/代码体**(delta 1);判断决策 / 精确值留人·PD / 详细设计定(大白话点名)。

### 3. 设计决策(决策块,跨需求综合;按需)

> 触发判定 / verbose 模板 / 四 gate / 降级 / 概念级边界全部见 `references/设计决策层-ADR速查.md`,这里只讲它在报告里怎么排。

- **何时有这章**:命中 design-smell(主数据归并等)或架构 fork + 过影响阈值时才有;纯定位小需求没有这章。
- **每个决策一个 ≤6 段决策块**(决策 / 为什么 / 方案比较 / 采纳 / 影响 / 兼容)。完整候选 / 被否 / 翻案 / 概念模型以此块 + 附录 C 降级表为准;三标签 / 推断链 / verbose 证据依赖表沉附录 / 速查,不在正文。
- 受该决策约束的落点归在附录 B 落点归属表(按落点名)。

### 4. 横切汇总

- **4.1 跨需求影响盘点**:受影响现有功能 / 方法 / 表(跨需求去重)+ 文档内部一致性自检(新构件名跨段对齐?现状段引的表真存在?否则写"待核实")+ **现状事实源一致性(D14)**:同一 `subject + predicate` 被多源给出不兼容值时,业务正文用"重要:...不一致(详见附录 C)"plain 醒目提示(`[冲突]` 记号沉附录),不静默二选一;多源一致不算冲突。
- **4.2 待确认 / 开放项与诚实边界**:RAG 无语料命中 / config `fallback_only` / 业务配置未物化 / Oracle 缺表 / `change_type` 粗标;人/PD 专属(枚举编码、索引命名、字段长度、验收、风险、工期)skill 不产。
- **4.3(可选)规模感**:至多按"改存量 N 处 + 新增 M 处"给体量,不产工期/风险。

## 溯源附录结构(固定 A/B/C/D)

- **附录 A:逐条出处(脚注定义区)**:每个 `[^aN]:` 一条 = 工具 + 关键参数 + 命中摘要(可重放);无工具命中写 `无工具命中,纯推断`。
- **附录 B:落点归属表(按落点名 key,无 L-NN)**:列 = 落点 | 出现于需求 | 归属决策 | conform | 证据。`conform` 固定三值枚举 `conform` / `翻案 -> 决策 M` / `gap/待确认`。每个落点 -> 归哪个决策 + conform + 证据(D10 内部一致保证;落点名本身是 key,比 L-NN 更有意义)。
- **附录 C:证据降级(只列被降级的 claim)**:`<claim> <- <依赖证据,如 search_source census 截断 5000 cap> -> [限制] <后果,如范围未闭合,③ 补>`。证据完整的不写;全完整写"无降级项"。与附录 A 不重(A = 原始来源,C = claim 级后果)。
- **附录 D:本次 MCP 调用清单**:工具 + 次数,供一眼确认 MCP 能力被真用。

## Worked example(中性合成 —— 假数据,不是真客户)

> 红线:worked example 全部用明显假的中性值(`com.example.app.*` / `APP_*`)。真客户跑出的报告只落 gitignored 路径,绝不进本文件。

### 0. 报告头

**输入来源**:`example-loyalty.eml`(合成需求,~7KB),分析日期 2026-06-24,目标 codebase `com.example.app`。
**工具态**:`health_check()` = engine `ok` / code_projection `ok` / oracle `connected` / jdt_ls `cold`(忽略) / models `ready`。全绿。`profile_info()` -> `rag_corpora=["app-biz-docs","app-ops-runbook"]`。
**怎么读**:业务正文是给业务方读的干净概要,不含任何尖括号记号;开放项直接写明"需谁确认 / 留哪个阶段定",矛盾用"重要:..."醒目提示。要逐条核验工具出处,看末尾溯源附录;MCP 调用清单见附录 D。
**recall 声明**:需求分解覆盖原文 3 个编号点(Phase 1 对照已核;§2.4 服务类型管理增强是额外 ADR 演示块,不计入这 3 点);影响落点是已定位证据子集,不保证完整召回。

### 1. 需求分解

会员积分(Loyalty Points)下 3 点:

1. `积分-1` 主账户给子账户共享积分额度,单次共享有上限。
2. `积分-2` 共享关系的建立 / 解除,记录绑定关系。
3. `积分-3` 共享流水查询页。

点间耦合:`积分-3` 依赖 `积分-1` 写出的划拨记录;`积分-2` 是 `积分-1` 划拨的前置校验数据源。
另:§2.4 + §3 额外演示一个补充点(服务类型管理增强),展示"主数据归并气味 -> 设计决策章 + 附录 B 落点归属",不计入上面 3 点。

### 2. 逐需求章

#### 块 2.1 — 积分-1:主账户给子账户共享积分额度,单次有上限

**当前系统现状分析**

现状总表:

| 现状结论 | 涉及对象 | 对设计的含义 |
|---|---|---|
| 积分余额只表达账户余额,无"共享/划拨"语义 | `APP_POINT_BALANCE` [^a1] | 余额表保持为余额事实源,划拨语义需另立概念承载 |
| 积分类型集中定义,当前无 SHARE/共享类型 | `PointTypeConst` [^a2] | 共享若需类型可在此扩,但不改余额表语义 |
| 扣减走业务规则入口而非通用 DAO | `PointDeductBC.doDeduct` [^a3] | 划拨是规则扩展点,应在 BC 层加分支,非改 DAO |

痛点:`doDeduct` 只扣本账户,无"扣主账户 / 记对端子账户"的划拨分支,也无单次上限配置 —— 现状满足不了主子账户划拨。

**新需求内容**

- 需求描述:主账户把积分额度划一部分给子账户,单笔有金额上限。
- 业务规则:(1) 扣主账户、记子账户;(2) 单次划拨 <= 上限;(3) 上限可配。
- 验收口径:原文未给出单次上限数值与失败回滚规则,需**需求方**补充确认。
- 业务语义:"共享额度"在本系统指主子账户间积分划拨,非营销赠送 [^a11]。

**概要设计(设计思路)**

- 推荐方向:在现有积分扣减规则旁新增"积分共享划拨"能力,保持余额表为余额事实源,新增共享 / 流水概念承载划拨语义。
- 为什么:现状只有余额扣减、无主子账户划拨语义;把共享硬塞进普通扣减会混淆业务含义。
- 范围意图:存量 `doDeduct` 作参照和必要扩展点;共享方法、单次上限配置、划拨流水由新增能力承载。
- 待定项:单次上限数值、失败回滚、流水保留策略,留**详细设计 / 产品**确定。

**影响落点**

- 改动现有:`PointDeductBC.doDeduct` 加"划拨"分支(扣主账户 / 记对端子账户)[^a3]。
- 新增:积分共享划拨方法(照 `PointDeductBC` 方法族惯例)[^a3];单次上限配置项 `points.share.max_per_txn`(现无)[^a15]。
- 不改但需兼容:余额表 `APP_POINT_BALANCE` 余额列语义不变,共享语义不污染余额事实源 [^a1]。
- 精确上限数值、回滚策略由**产品 / PD**确定,本阶段不产。

#### 块 2.2 — 积分-2:共享关系建立 / 解除(全新增需求示范)

**当前系统现状分析**

现状总表:

| 现状结论 | 涉及对象 | 对设计的含义 |
|---|---|---|
| 未定位到"主子账户积分共享绑定"能力 | `APP_POINT_SHARE`(零命中)[^a4] | 绑定关系需全新增,无存量可改 |
| 存量只有通用账户绑定,非积分共享 | `AcctBindBC` [^a5] | 可照其 bind/unbind 模式拟,但不能冒充业务事实源 |

痛点:现状无可维护的共享绑定关系,积分-1 无法判断主子账户是否允许共享。

**新需求内容**

- 需求描述:建立 / 解除主子账户积分共享绑定,留绑定关系记录。
- 业务规则:(1) 一条绑定 = 主账户 + 子账户 + 状态;(2) 解除是软删 / 置失效。
- 验收口径:原文未给,需**需求方**补充。
- 业务语义:本系统业务文档未见"积分共享绑定"专门定义(RAG 未命中,不等于不存在,详见溯源附录)[^a12]。

**概要设计(设计思路)**

- 推荐方向:新增积分共享绑定关系,独立承载主账户、子账户、状态、审计等概念语义。
- 为什么:现状通用账户绑定不是积分共享绑定,不能直接冒充业务事实源。
- 范围意图:提供绑定 / 解绑维护能力,作为积分-1 划拨前置校验数据源。
- 待定项:状态枚举、字段长度、是否允许多子账户绑定,留**详细设计**确定。

**影响落点**

- 改动现有:无(本需求全新增;现状已证 `PointShare*` 零命中)。
- 新增:积分共享绑定表 `APP_POINT_SHARE`(照存量 `APP_POINT_BALANCE` 命名 / 审计列惯例,不列 DDL)[^a4];绑定 / 解绑方法(照 `AcctBindBC` 的 bind/unbind 模式)[^a5]。
- 不改但需兼容:存量通用账户绑定 `AcctBindBC` 语义不变,新绑定不复用其表 [^a5]。
- 表列名、字段长度、状态枚举由**产品 / PD**在详细设计阶段确定。

### 3. 设计决策(决策块)

> 积分-1/2/3 是单点定位,不触发;块 2.4 命中主数据归并气味,触发 决策 1。verbose 候选 / 推断链 / 证据依赖表见 `references/设计决策层-ADR速查.md` + 附录 C。

#### 决策 1:服务类型归一为显式主数据,而非各消费方散落补丁

- **决策**:新增统一"服务类型主数据"实体(显式 分类维度 / 生命周期状态 / 审计),把散落在前缀判断里的隐式分类收口;旧 `APP_SERVICE_TYPE_DICT` 迁入并保留兼容。
- **为什么**:服务类型现落静态字典 [^a7]、分类靠前端前缀散落判断 [^a8]、多消费方读(含字符串派发)[^a9][^a10] —— 字典已在隐式承担主数据角色,但分类逻辑被前缀绑架、散落各消费方。
- **方案比较**:① 统一主数据实体 + 迁字典 + 各消费方读点替换(收口);② 在 `APP_SERVICE_TYPE_DICT` 加列(扩旧,分类仍部分靠前缀);③ 每消费方各自加补丁表 / 字段(散落)。
- **采纳**:选 ①。方案 ③ 放大跨消费方一致性债;方案 ② 不解决"分类逻辑散落",治标不治本;只有统一主数据 + 显式分类字段能收口。
- **影响**:改各消费方读点(改读主数据);新增主数据实体 + 生命周期状态 + 显式分类字段 + 维护接口 / 维护页 Tab。落点归属见附录 B。
- **兼容/翻案条件**:旧 `TYPE_CODE` 前缀编码不得废(存量数据 + 外部对接按它),字典对外只读视图保留;若 `APP_SERVICE_TYPE_DICT` 实为上游主数据视图、或消费方 census 收敛后 <=2 个读点 -> 改选方案 ②。
